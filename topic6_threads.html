1. What Are Threads?

Ένα thread (νήμα) είναι μια ελαφριά ροή εκτέλεσης μέσα σε μια διεργασία. Ενώ μια διεργασία (process) είναι μια "βαριά" οντότητα με δικό της address space, τα threads μοιράζονται τον ίδιο χώρο διευθύνσεων και τα ίδια ανοιχτά αρχεία. Κάθε thread έχει μόνο τα δικά του: Program Counter, Stack, Registers.

Thread = Lightweight Process (LWP)
  • Shared: Address space (code, data, heap), open file descriptors, signal handlers, current directory, user/group IDs.
  • Private (per-thread): Thread ID, Program Counter, registers, stack, errno, signal mask, scheduling priority.
  • Key Insight: Μπορούν να τρέξουν παράλληλα σε πολλαπλούς πυρήνες, χωρίς το overhead του fork() (δεν αντιγράφεται address space).

Tip — Thread vs Process (quick mental model)
Σκέψου μια process σαν ένα σπίτι (δικός του χώρος, δικές του πόρτες, δικό του κλειδί).
Τα threads είναι οι ένοικοι — μοιράζονται κουζίνα, σαλόνι, μπάνιο (shared memory), αλλά ο καθένας έχει δικό του κρεβάτι (stack) και δική του λίστα υποχρεώσεων (PC).
POSIX Threads (pthreads)

Το πρότυπο POSIX.1c (IEEE Std 1003.1c-1995) ορίζει το pthreads API:

  • #include <pthread.h> — header αρχείο
  • Compile: gcc prog.c -lpthread ή -pthread
  • Πάνω από 60 συναρτήσεις στο API
  • Τύπος thread: pthread_t (opaque type)
Compilation Warning
Αν ξεχάσεις το -lpthread, ο compiler δεν θα βρει τις pthread_ συναρτήσεις. Πάντα link με -lpthread!
Two-Level Thread Model

Στο Linux, τα user-level threads αντιστοιχούν σε kernel threads μέσω Lightweight Processes (LWP). Ο kernel scheduler βλέπει τα LWP, όχι τα user threads. Αυτό σημαίνει ότι τα threads μπορούν να τρέχουν σε διαφορετικούς πυρήνες (true parallelism).

2. Thread vs Process Comparison

Γιατί να χρησιμοποιήσουμε threads αντί για fork(); Η δημιουργία thread είναι 10-100x ταχύτερη από fork(), δεν γίνεται αντιγραφή address space, και η επικοινωνία γίνεται μέσω shared memory αντί pipes. Αλλά υπάρχουν και σοβαρά μειονεκτήματα.

Thread Advantages
  • Γρηγορότερη δημιουργία/τερματισμός
  • Εύκολη κοινή χρήση δεδομένων (shared address space)
  • Λιγότεροι πόροι (δεν γίνεται copy address space)
  • Φυσική παραλληλοποίηση σε multi-core CPUs
Thread Dangers
  • Κοινή μνήμη = Race conditions, data corruption
  • Ένα thread crash = ολόκληρη η διεργασία πεθαίνει
  • Δύσκολο debugging (non-deterministic bugs)
  • Απαιτεί synchronization primitives (mutexes, κλπ.)
🧠
Exam Tip — Πότε threads, πότε processes;
Threads: Όταν χρειάζεσαι γρήγορη κοινή πρόσβαση σε δεδομένα (π.χ. web server, matrix multiplication).
Processes: Όταν θέλεις isolation (π.χ. ένα service που δεν πρέπει να επηρεάσει άλλα αν crashαρει).
Κλασική ερώτηση εξέτασης: "Αν ένα thread κάνει segfault, τι γίνεται;" — Ολόκληρη η διεργασία τερματίζεται!

3. Creating Threads: pthread_create & pthread_exit

Η pthread_create() δημιουργεί νέο thread. Το νέο thread αρχίζει εκτέλεση από μια thread function (σαν τη main() για processes). Η pthread_exit() τερματίζει μόνο το τρέχον thread, χωρίς να σκοτώσει τη διεργασία.

pthread_create()
int pthread_create( pthread_t *thread, // output: thread ID const pthread_attr_t *attr, // NULL = defaults void *(*start_routine)(void *), // thread function void *arg // argument to function );
0
Success
> 0
Error code (NOT -1 like fork!)
Thread Function Signature
Πρέπει πάντα: void *function_name(void *arg). Δέχεται ένα void * argument και επιστρέφει void *.
pthread_exit()
void pthread_exit(void *retval); // Τερματίζει μόνο το ΤΡΕΧΟΝ thread // Δεν τερματίζει ολόκληρη τη διεργασία! // Η retval μπορεί να ληφθεί μέσω pthread_join()
exit() vs pthread_exit()
Η exit() τερματίζει ολόκληρη τη διεργασία (και όλα τα threads). Η pthread_exit() τερματίζει μόνο το τρέχον thread. Αν η main() τερματίσει με return, η διεργασία πεθαίνει — αν θέλεις τα υπόλοιπα threads να συνεχίσουν, χρησιμοποίησε pthread_exit() στη main!
Example — multi_hello.c
#include <stdio.h> #include <stdlib.h> #include <pthread.h> #define NUM_THREADS 5 void *hello(void *argp) { int my_id = *(int *)argp; printf("Thread %d says Hello!\n", my_id); pthread_exit(NULL); } int main() { pthread_t tid[NUM_THREADS]; int id[NUM_THREADS]; for (int i = 0; i < NUM_THREADS; i++) { id[i] = i; pthread_create(&tid[i], NULL, hello, &id[i]); } for (int i = 0; i < NUM_THREADS; i++) pthread_join(tid[i], NULL); return 0; }
Non-deterministic Output!
Η σειρά εκτύπωσης αλλάζει κάθε φορά! Π.χ.: Thread 2, Thread 0, Thread 4, Thread 1, Thread 3. Ο scheduler αποφασίζει ποιο thread τρέχει πότε.
Common Bug — Passing Shared Variable
// ❌ WRONG: all threads may see i=5! for (i = 0; i < 5; i++) pthread_create(&tid[i], NULL, hello, &i); // ✅ CORRECT: each thread gets its own copy for (i = 0; i < 5; i++) { id[i] = i; pthread_create(&tid[i], NULL, hello, &id[i]); }
Αν περάσεις &i (τη loop variable), η τιμή μπορεί να αλλάξει πριν το thread την διαβάσει! Χρησιμοποίησε ξεχωριστό array.

4. Thread Management: join, detach, self, cancel, kill

Αφού δημιουργήσεις ένα thread, πρέπει να το διαχειριστείς: να περιμένεις να τελειώσει (join), να το αφήσεις ελεύθερο (detach), ή ακόμα και να το ακυρώσεις (cancel). Αν δεν κάνεις ούτε join ούτε detach, δημιουργείται resource leak.

pthread_join() — Wait for thread
int pthread_join(pthread_t thread, void **retval); // Blocks until thread terminates // retval receives the exit value // Σαν τη wait() για processes!

Αν δεν σε νοιάζει η τιμή επιστροφής, πέρνα NULL ως δεύτερο argument. Κάθε thread μπορεί να γίνει join μόνο μία φορά.

pthread_detach() — Fire and forget
int pthread_detach(pthread_t thread); // Resources freed automatically on termination // Cannot join a detached thread!

Χρήσιμο για "background" threads. Δεν χρειάζεται κάποιος να τα περιμένει. Όταν τελειώσουν, καθαρίζονται αυτόματα.

pthread_self() — Get own ID
pthread_t pthread_self(void); // Returns the calling thread's ID // Like getpid() for threads
pthread_cancel() — Request cancellation
int pthread_cancel(pthread_t thread); // Sends cancellation request // Thread may not die immediately! // Depends on cancellation state/type

Η ακύρωση δεν είναι instant — το thread ελέγχει σε cancellation points (π.χ. μέσα σε pthread_join, read, sleep). Μπορεί να απενεργοποιηθεί.

pthread_kill() — Send signal to thread
int pthread_kill(pthread_t thread, int sig); // Like kill() but for a specific thread // sig=0 tests if thread exists
🧠
Tip — join vs detach
Πρέπει να κάνεις ένα από τα δύο για κάθε thread (join ή detach). Αν δεν κάνεις κανένα, τα resources δεν ελευθερώνονται = thread leak (σαν zombie processes).
Κανόνας: Χρειάζεσαι αποτέλεσμα → join. Δεν σε νοιάζει → detach.
⚠️

5. Error Handling & Safe Return Values

Οι pthread functions δεν χρησιμοποιούν errno! Επιστρέφουν 0 για επιτυχία και θετικό αριθμό (error code) για σφάλμα. Χρησιμοποίησε strerror() αντί perror().

Error Handling Pattern
int err; err = pthread_create(&tid, NULL, func, arg); if (err != 0) { fprintf(stderr, "create: %s\n", strerror(err)); exit(1); } // ❌ WRONG: perror("create"); — αυτή χρησιμοποιεί errno // ❌ WRONG: if (err == -1) — δεν επιστρέφουν -1!
strerror() vs perror()
Η perror() ελέγχει το global errno. Οι pthread functions δεν θέτουν errno — επιστρέφουν τον error code απευθείας. Χρησιμοποίησε strerror(err) για να μετατρέψεις τον κωδικό σε string.
Safe Return Values from Threads

Τι μπορεί να επιστρέψει ένα thread μέσω pthread_exit();

✅ Safe
  • malloc'd memory: int *p = malloc(sizeof(int)); *p = 42; pthread_exit(p);
  • String literals: pthread_exit("done"); — ζουν στο .rodata segment
  • Global variables: Ζουν στο data segment (αλλά πρόσεξε concurrent access!)
❌ Unsafe
  • Automatic (local) variables: int x = 42; pthread_exit(&x); — Το stack του thread καταστρέφεται μετά το exit! Dangling pointer!
🧠
Exam Tip — Thread Return Memory Safety
Μην επιστρέφεις ποτέ δείκτη σε τοπική (automatic) μεταβλητή από ένα thread. Όταν το thread τερματίζει, το stack frame του καταστρέφεται και η μνήμη αποδεσμεύεται. Ο δείκτης γίνεται dangling pointer, και οποιαδήποτε προσπάθεια πρόσβασης σε αυτόν από το thread-caller (μέσω pthread_join) θα οδηγήσει σε garbage values ή Segmentation Fault.
Ασφαλείς τρόποι επιστροφής:
  • Δέσμευση στο heap μέσω malloc() (ο caller αναλαμβάνει το free()).
  • Χρήση static ή global μεταβλητών (με προσοχή στο synchronization).
  • Casting της ίδιας της τιμής σε void* (αν χωράει, π.χ. pthread_exit((void*)(intptr_t)x)) χωρίς pointer reference.

6. Race Conditions

Ένα race condition συμβαίνει όταν δύο ή περισσότερα threads προσπελαύνουν τα ίδια δεδομένα ταυτόχρονα και τουλάχιστον ένα γράφει. Το αποτέλεσμα εξαρτάται από τη σειρά εκτέλεσης (scheduling) — η οποία δεν είναι ντετερμινιστική.

Example — wordcount1.c (Buggy!)
int total_words = 0; // GLOBAL — shared by all threads! void *count_words(void *arg) { char *filename = (char *)arg; int count = 0; // ... count words in file ... total_words += count; // ⚠️ RACE CONDITION! pthread_exit(NULL); }
The Problem
Η εντολή total_words += count δεν είναι atomic! Αποτελείται από 3 βήματα:
1. LOAD total_words → register
2. ADD count → register
3. STORE register → total_words
Αν δύο threads κάνουν αυτά τα βήματα ταυτόχρονα, μια πρόσθεση μπορεί να χαθεί!
Race Condition Visualization
total_words = 100 αρχικά
Thread A: LOAD 100 → regA
Thread B:            LOAD 100 → regB
Thread A: ADD 5 → 105
Thread B:            ADD 3 → 103
Thread A: STORE 105 → total_words
Thread B:            STORE 103 → total_words
Result: 103 (should be 108!) ← Thread A's work LOST!
🧠
Tip — "Correct" δεν σημαίνει σωστό
Ένα πρόγραμμα με race condition μπορεί να φαίνεται ότι δουλεύει σωστά 99% του χρόνου. Αυτό κάνει τα race conditions εξαιρετικά δύσκολο να βρεθούν. Μπορεί να εμφανιστούν μόνο κάτω από φόρτωση, ή σε συγκεκριμένα hardware. Αν κοινά δεδομένα γράφονται από πάνω από ένα thread, χρειάζεσαι synchronization.

7. POSIX Mutexes (Mutual Exclusion)

Ένας mutex (MUTual EXclusion) είναι ένα κλείδωμα. Μόνο ένα thread τη φορά μπορεί να κρατάει ένα mutex. Αν ένα δεύτερο thread προσπαθήσει να κλειδώσει, θα μπλοκάρει μέχρι ο πρώτος να ξεκλειδώσει. Αυτός είναι ο βασικός τρόπος προστασίας critical sections.

Mutex API
// Initialization (two ways) pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER; // static pthread_mutex_init(&mtx, NULL); // dynamic // Lock / Unlock pthread_mutex_lock(&mtx); // blocks if locked pthread_mutex_trylock(&mtx); // returns EBUSY if locked pthread_mutex_unlock(&mtx); // release // Destruction pthread_mutex_destroy(&mtx); // free resources
Static vs Dynamic Init
PTHREAD_MUTEX_INITIALIZER για global/static mutexes. pthread_mutex_init() αν ο mutex είναι δυναμικός ή χρειάζεσαι non-default attributes.
Critical Section Pattern
pthread_mutex_lock(&mtx); /* ── Critical Section ── */ /* Μόνο ΕΝΑ thread μπορεί να είναι εδώ τη φορά! */ total_words += count; /* ── End Critical Section ── */ pthread_mutex_unlock(&mtx);
wordcount2.c — Fixed with Mutex
int total_words = 0; pthread_mutex_t count_mutex = PTHREAD_MUTEX_INITIALIZER; void *count_words(void *arg) { char *filename = (char *)arg; int count = 0; // ... count words in file ... pthread_mutex_lock(&count_mutex); // 🔒 total_words += count; // safe! pthread_mutex_unlock(&count_mutex); // 🔓 pthread_exit(NULL); }
Fixed!
Τώρα μόνο ένα thread τη φορά μπορεί να τροποποιήσει το total_words. Το αποτέλεσμα είναι πάντα σωστό.
Mutex Rules
  • Μόνο ο κάτοχος μπορεί να κάνει unlock
  • Κάνε lock μόνο στο μικρότερο δυνατό section κώδικα
  • Πάντα ξεκλείδωνε τον mutex — αν ξεχάσεις = deadlock
  • trylock() επιστρέφει EBUSY αν κλειδωμένο — δεν μπλοκάρει

8. Condition Variables

Τα condition variables επιτρέπουν σε ένα thread να περιμένει μέχρι κάποια συνθήκη γίνει αληθής, χωρίς busy waiting. Δουλεύουν πάντα σε συνδυασμό με ένα mutex. Σκέψου τα ως "alarm" — αντί να ελέγχεις κάθε λίγο, κοιμάσαι και σε ξυπνάνε.

Condition Variable API
// Initialization pthread_cond_t cv = PTHREAD_COND_INITIALIZER; // static pthread_cond_init(&cv, NULL); // dynamic // Wait — releases mutex, sleeps, re-acquires mutex on wakeup pthread_cond_wait(&cv, &mtx); // Signal — wake ONE waiting thread pthread_cond_signal(&cv); // Broadcast — wake ALL waiting threads pthread_cond_broadcast(&cv); // Destruction pthread_cond_destroy(&cv);
How pthread_cond_wait() Works
pthread_cond_wait(&cv, &mtx):
1. Unlock mutex (ατομικά)
2. Sleep — thread μπαίνει σε αναμονή
3. Wake up (σήμα ελήφθη)
4. Re-lock mutex (μπλοκάρει αν χρειαστεί)
5. Return — ο caller ξανακρατάει τον mutex
Atomic unlock+sleep
Τα βήματα 1+2 γίνονται ατομικά — κανένα thread δεν μπορεί να στείλει signal μεταξύ τους. Αυτό αποτρέπει lost wakeups.
Example — mutex_condvar.c
pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER; pthread_cond_t cv = PTHREAD_COND_INITIALIZER; int ready = 0; void *waiter(void *arg) { pthread_mutex_lock(&mtx); while (!ready) // ← WHILE, not IF! pthread_cond_wait(&cv, &mtx); printf("Ready! Proceeding...\n"); pthread_mutex_unlock(&mtx); return NULL; } void *signaler(void *arg) { sleep(2); pthread_mutex_lock(&mtx); ready = 1; printf("Signaling...\n"); pthread_cond_signal(&cv); pthread_mutex_unlock(&mtx); return NULL; }
counter.c — Threshold Signaling

3 threads αυξάνουν ένα counter. Ένα 4ο thread κοιμάται μέχρι ο counter φτάσει ένα threshold:

// Incrementer thread pthread_mutex_lock(&count_mutex); count++; if (count == COUNT_LIMIT) pthread_cond_signal(&count_threshold_cv); pthread_mutex_unlock(&count_mutex); // Watcher thread pthread_mutex_lock(&count_mutex); while (count < COUNT_LIMIT) pthread_cond_wait(&count_threshold_cv, &count_mutex); printf("Threshold reached! count=%d\n", count); pthread_mutex_unlock(&count_mutex);

9. Monitor-Style Programming & Spurious Wakeups

Ο monitor-style programming είναι η σωστή μεθοδολογία: mutexes + condition variables, με while loops γύρω από τα wait calls. Αυτό ονομάζεται "defensive programming" γιατί προστατεύει από spurious wakeups και concurrent races.

Spurious Wakeups — Γιατί while αντί if;
❌ if (wrong)
if (!ready) // check once pthread_cond_wait(&cv, &m); // may proceed when not ready!
✅ while (correct)
while (!ready) // re-check pthread_cond_wait(&cv, &m); // guaranteed ready == true!

Γιατί; Ένα thread μπορεί να ξυπνήσει χωρίς λόγο (spurious wakeup) — είτε από τον scheduler, είτε γιατί κάποιο άλλο thread "έφαγε" τη συνθήκη μεταξύ signal και re-lock. Με while, ξαναελέγχεις τη συνθήκη μετά το wakeup.

🧠
Exam Tip — Spurious Wakeups & Condition Variables
Η χρήση της while γύρω από την pthread_cond_wait() είναι υποχρεωτική για δύο λόγους:
  • Spurious Wakeup: Ο scheduler του λειτουργικού συστήματος μπορεί να ξυπνήσει ένα μπλοκαρισμένο thread χωρίς να έχει σταλεί signal (συμβαίνει για λόγους απόδοσης και χαμηλού επιπέδου αρχιτεκτονικής).
  • Mesa/Lampson Semantics: Όταν ένα thread λαμβάνει signal και ξυπνάει, δεν εκτελείται αμέσως. Πρέπει πρώτα να ξανα-αποκτήσει το mutex. Στο ενδιάμεσο διάστημα, ένα άλλο thread μπορεί να μπει, να αποκτήσει το mutex, να καταναλώσει τον πόρο (ready = false) και να βγει. Αν είχαμε if, το ξυπνημένο thread θα συνέχιζε τυφλά και θα προσπελαύανε μη έγκυρο state.
Abstract Monitor Operations
OperationEffect
lock(m)Acquire mutex. Block if already held.
unlock(m)Release mutex.
wait(cv, m)Atomically: unlock m, sleep. On wakeup: re-lock m.
signal(cv)Wake one waiter (if any).
broadcast(cv)Wake all waiters.
Golden Rules of Monitor-Style Programming
  1. Πάντα κλείδωνε τον mutex πριν ελέγξεις ή αλλάξεις τη shared μεταβλητή
  2. Πάντα while loop γύρω από pthread_cond_wait()
  3. Σηματοδότησε (signal/broadcast) ενώ κρατάς τον mutex (ή αμέσως μετά το unlock)
  4. Κράτα τα critical sections μικρά — ελαχιστοποίησε τον χρόνο κλειδώματος
  5. Ένας mutex ανά condition variable — μην αναμιγνύεις mutexes
Readers/Writers Pattern

Πολλαπλοί readers ταυτόχρονα ΟΚ, αλλά μόνο ένας writer:

// Reader lock(m); while (active_writers > 0) wait(cv, m); active_readers++; unlock(m); /* ... read ... */ lock(m); active_readers--; if (active_readers == 0) signal(cv); unlock(m); // Writer lock(m); while (active_writers + active_readers > 0) wait(cv, m); active_writers++; unlock(m); /* ... write ... */ lock(m); active_writers--; broadcast(cv); unlock(m);

10. Thread Safety & Reentrant Functions

Μια συνάρτηση είναι thread-safe αν μπορεί να κληθεί ταυτόχρονα από πολλαπλά threads χωρίς race conditions. Πολλές κλασικές C functions δεν είναι thread-safe γιατί χρησιμοποιούν static εσωτερικά buffers.

Thread-Unsafe Functions
UnsafeThread-safe Alternative
strtok()strtok_r()
gethostbyname()gethostbyname_r()
localtime()localtime_r()
rand()rand_r()
ctime()ctime_r()
readdir()readdir_r()
The _r Suffix Convention
Οι reentrant εκδόσεις τελειώνουν σε _r (reentrant). Δέχονται ένα εξωτερικό buffer αντί να χρησιμοποιούν internal static storage.
Making a Function Thread-Safe

3 στρατηγικές:

  1. Replace static with caller-provided buffer: Η strtok_r() δέχεται char **saveptr αντί internal state.
  2. Lock with mutex: Wrap non-safe calls με mutex. Απλό αλλά μειώνει parallelism.
  3. Thread-Local Storage: Κάθε thread έχει δικό του αντίγραφο (TLS).
🧠
Exam Tip — Γιατί η strtok() δεν είναι thread-safe;
Η strtok() χρησιμοποιεί ένα internal static pointer για να θυμάται πού σταμάτησε. Αν δύο threads καλέσουν strtok() ταυτόχρονα, ο ένας θα "πατήσει" τον pointer του άλλου. Η strtok_r() σπάει αυτό βάζοντας τον caller να περάσει τον saveptr.

11. Producer-Consumer Problem

Το Producer-Consumer είναι κλασικό πρόβλημα συγχρονισμού: ένας (ή περισσότεροι) producers βάζουν items σε ένα bounded buffer, και ένας (ή περισσότεροι) consumers τα αφαιρούν. Ο producer πρέπει να περιμένει αν ο buffer είναι γεμάτος, και ο consumer αν είναι άδειος.

Bounded Cyclical Buffer
#define BUFSIZE 8 typedef struct { int data[BUFSIZE]; int start; // next position to read from int end; // next position to write to int count; // current number of items } pool_t; void place(pool_t *pool, int data) { pthread_mutex_lock(&mtx); while (pool->count == BUFSIZE) // full? pthread_cond_wait(&cond_nonfull, &mtx); pool->data[pool->end] = data; pool->end = (pool->end + 1) % BUFSIZE; pool->count++; pthread_cond_signal(&cond_nonempty); // wake consumer pthread_mutex_unlock(&mtx); } int obtain(pool_t *pool) { pthread_mutex_lock(&mtx); while (pool->count == 0) // empty? pthread_cond_wait(&cond_nonempty, &mtx); int data = pool->data[pool->start]; pool->start = (pool->start + 1) % BUFSIZE; pool->count--; pthread_cond_signal(&cond_nonfull); // wake producer pthread_mutex_unlock(&mtx); return data; }
prod-cons.c — Full Program
pthread_mutex_t mtx; pthread_cond_t cond_nonempty, cond_nonfull; pool_t pool; void *producer(void *ptr) { while (num_of_items > 0) { place(&pool, num_of_items); printf("producer: %d\n", num_of_items); num_of_items--; pthread_cond_signal(&cond_nonempty); usleep(300000); } pthread_exit(0); } void *consumer(void *ptr) { while (num_of_items > 0 || pool.count > 0) { printf("consumer: %d\n", obtain(&pool)); pthread_cond_signal(&cond_nonfull); usleep(500000); } pthread_exit(0); } int main() { pthread_t cons, prod; initialize(&pool); pthread_mutex_init(&mtx, 0); pthread_cond_init(&cond_nonempty, 0); pthread_cond_init(&cond_nonfull, 0); pthread_create(&cons, 0, consumer, 0); pthread_create(&prod, 0, producer, 0); pthread_join(prod, 0); pthread_join(cons, 0); pthread_cond_destroy(&cond_nonempty); pthread_cond_destroy(&cond_nonfull); pthread_mutex_destroy(&mtx); return 0; }

Food for Thought
Q: Αν ο usleep αντικατασταθεί με usleep(0), τι αλλάζει;
A: Ο producer τρέχει πολύ πιο γρήγορα, γεμίζει τον buffer, εμφανίζονται "Found Buffer Full" μηνύματα. Η σχεδίαση με condition variables αποτρέπει busy-waiting.

Q: Τι δεδομένα προστατεύονται από τον mutex;
A: Ο buffer (pool), count, start, end. Η μεταβλητή num_of_items ΔΕΝ προστατεύεται — αυτό είναι bug αν υπάρχουν πολλοί producers!

12. Concurrency Errors

Οι κατηγορίες σφαλμάτων σε multi-threaded κώδικα. Κάθε τύπος σφάλματος απαιτεί διαφορετική αντιμετώπιση. Η κατανόηση αυτών είναι κρίσιμη για εξετάσεις.

1. Data Races
Definition
Δύο ή περισσότερα threads προσπελαύνουν τα ίδια δεδομένα ταυτόχρονα, τουλάχιστον ένα γράφει, και δεν υπάρχει synchronization.

Fix: Mutex γύρω από κάθε shared variable access.

2. Atomicity Violations
Definition
Μια σειρά λειτουργιών που πρέπει να εκτελεστεί ατομικά, αλλά κάποιο thread παρεμβαίνει μεταξύ τους.
// Thread A: check + use must be atomic! if (ptr != NULL) // ← Thread B sets ptr=NULL here! ptr->field = 42; // ← CRASH: null dereference

Fix: Wrap check-then-act σε ένα mutex.

3. Order (Logical) Violations
Definition
Η εκτέλεση πρέπει να ακολουθεί μια λογική σειρά (π.χ. init πριν use), αλλά δεν εξασφαλίζεται.

Fix: Condition variables για ordering guarantees.

4. Deadlocks
Definition
Δύο ή περισσότερα threads περιμένουν αμοιβαία — κανένα δεν μπορεί να προχωρήσει. Κλασική κατάσταση: Thread A κρατά Lock 1 και θέλει Lock 2, Thread B κρατά Lock 2 και θέλει Lock 1.
Thread A: lock(m1) → ... → lock(m2) BLOCKED
Thread B: lock(m2) → ... → lock(m1) BLOCKED
DEADLOCK! Κανένα δεν μπορεί να συνεχίσει.

Fix: Πάντα κλείδωνε τα mutexes με σταθερή σειρά (lock ordering). Ή χρησιμοποίησε trylock().

🧠
Mnemonic — DRAO (Data Race, Atomicity, Ordering, Deadlock)
Θυμήσου τα 4 είδη: Data race (no lock), R... Atomicity violation (lock but not around entire operation), Order violation (wrong sequence), Deadlock (circular wait). Η λύση: περισσότερα locks + σωστή σειρά.
⚙️

13. Locking Strategies & Composability

Πόσο "λεπτό" πρέπει να είναι το locking; Coarse-grained = ένα lock για τα πάντα (απλό αλλά αργό). Fine-grained = πολλά locks (γρήγορο αλλά πολύπλοκο και κίνδυνος deadlock). Η επιλογή εξαρτάται από τις ανάγκες.

Why Multi-Threaded Code Is Hard
  • Locks are global: Η ιδιότητα "ποιο lock κρατώ" επηρεάζει ολόκληρο το πρόγραμμα — δεν κρύβεται πίσω από interface.
  • No modularity: Callers πρέπει να ξέρουν τι locks παίρνουν τα callees.
  • Condition variables are non-local: Πρέπει να ξέρεις ποια CV να σηματοδοτήσεις κάθε φορά.
  • No composability: Δεν μπορείς να φτιάξεις safe services από safe sub-services!
Account Library — Composability Problem
typedef struct { int balance; pthread_mutex_t account_mutex; } account_type; void synch_withdraw(account_type *acc, int amount) { lock(acc->account_mutex); withdraw(acc, amount); unlock(acc->account_mutex); } // Client code — ATOMICITY VIOLATION! void move(account_type *acc1, account_type *acc2, int amount) { synch_withdraw(acc1, amount); // lock1, withdraw, unlock1 // ← state visible between operations! synch_deposit(acc2, amount); // lock2, deposit, unlock2 }
Problem 1: Atomicity
Μεταξύ withdraw και deposit, κάποιος μπορεί να δει ότι χρήματα "εξαφανίστηκαν".
// Fix attempt — DEADLOCK RISK! void atomic_move(account_type *acc1, account_type *acc2, int amt) { lock(acc1->account_mutex); lock(acc2->account_mutex); // DEADLOCK if another withdraw(acc1, amt); // thread does move(acc2,acc1)! deposit(acc2, amt); unlock(acc2->account_mutex); unlock(acc1->account_mutex); } // move(s,t,...) || move(t,s,...) → DEADLOCK // move(s,s,...) → SELF-DEADLOCK!
Problem 2: Deadlock
Αν move(A,B) τρέχει ταυτόχρονα με move(B,A), κάθε ένα κλειδώνει τον πρώτο λογαριασμό και περιμένει τον δεύτερο → deadlock. Λύση: lock ordering (πάντα κλείδωνε τον λογαριασμό με τη μικρότερη διεύθυνση πρώτα).

1. Πώς Μοιάζει η Εξέταση

Η εξέταση έχει τυπικά σταθερή δομή: ~5 θέματα σε 120 λεπτά, χωρίς σημειώσεις/βιβλία, με γραπτό κώδικα στο χαρτί. Πέρα από τη θεωρία, τα θέματα κώδικα ανήκουν σχεδόν πάντα στις ίδιες 4 «οικογένειες» ασκήσεων — και αυτές δουλεύουμε σε αυτή τη σελίδα.

Οικογένεια Περιεχόμενο
Θεωρία Σύντομα ερωτήματα «περιγράψτε 2 διαφορές / 2 τρόπους / 2 περιπτώσεις»
Sockets Συνάρτηση C που διαβάζει/γράφει δυαδικά δεδομένα σε socket (read loop + byte order), συχνά με mini ερώτημα «τι τιμή/πόσα bytes;»
Threads Πρόγραμμα με N threads συγχρονισμένα με mutex + condition variable (χωρίς sleep / busy waiting)
Processes Πρόγραμμα με fork() + exec*() + wait()
Bash Script με έλεγχο παραμέτρων, έλεγχο αρχείων/καταλόγων και grep/wc
📎
Στο φύλλο της εξέτασης δίνονται prototypes
Το φύλλο συνήθως κλείνει με prototypes των συναρτήσεων που μπορεί να χρειαστείς: fork, execvp, wait, waitpid, kill, read, write, ntohs/htons/ntohl/htonl, pthread_create/join, pthread_mutex_*, pthread_cond_*. Δεν χρειάζεται να θυμάσαι απ' έξω υπογραφές — χρειάζεται να ξέρεις πώς συνδυάζονται.
Στρατηγική 120 λεπτών
  • Πρώτα η θεωρία: 10–15 λεπτά, είναι «δώρο» — δες την Θεωρία-Flash παρακάτω.
  • Μετά το θέμα που σου κάθεται καλύτερα. Τα θέματα κώδικα είναι ανεξάρτητα μεταξύ τους.
  • Σκελετός πριν τον κώδικα: γράψε πρώτα σε σχόλια τα βήματα (π.χ. «1. διάβασε n, 2. loop, 3. στείλε άθροισμα») και γέμισε μετά.
  • Μην κολλάς σε λεπτομέρειες σύνταξης — ένα ξεχασμένο ; δεν κοστίζει όσο μια λάθος λογική συγχρονισμού.
  • Πρόσεξε τα «Απαραίτητα»: αν η εκφώνηση λέει «χωρίς busy waiting» και βάλεις sleep() σε loop, χάνεις το θέμα.
Ο κανόνας του καθηγητή
«Αν γράψατε τον κώδικα των εργασιών μόνοι σας, θα είστε μια χαρά.» Οι ασκήσεις είναι μικρογραφίες των εργασιών: η άσκηση socket είναι το πρωτόκολλο επικοινωνίας που υλοποίησες, η άσκηση threads η ουρά εργασιών με τους workers, η άσκηση fork/exec η εκτέλεση των jobs. Δες τη γέφυρα εργασιών → εξέτασης.

2. Θεωρία-Flash

Τα ερωτήματα θεωρίας ζητούν σχεδόν πάντα σύντομες, ονομαστικές απαντήσεις: «2 διαφορές», «2 τρόποι», «2 περιπτώσεις». Παρακάτω είναι τα κλασικά ζεύγη με έτοιμες απαντήσεις σε «exam length» — δύο γραμμές ανά σκέλος αρκούν.

Process vs Thread — 2 διαφορές
  • Address space: κάθε process έχει δικό του· τα threads μιας process μοιράζονται τον ίδιο (code, data, heap, fds). Κάθε thread έχει μόνο δικό του stack/PC/registers.
  • Κόστος δημιουργίας: το fork() αντιγράφει ολόκληρη εικόνα process· το pthread_create() είναι 10–100× φθηνότερο.
  • Bonus εναλλακτική: επικοινωνία — threads μέσω κοινής μνήμης απευθείας, processes μόνο μέσω IPC (pipes, sockets, shared memory).
2 περιπτώσεις που ένα process λαμβάνει SIGCHLD
  • Όταν ένα παιδί του τερματίσει (κανονικά με exit ή από σήμα).
  • Όταν ένα παιδί του σταματήσει (π.χ. SIGSTOP/SIGTSTP) ή συνεχίσει (SIGCONT).
Blocking vs Non-blocking I/O
  • Blocking: η read() «παγώνει» την κλήση μέχρι να υπάρξουν δεδομένα (ή EOF).
  • Non-blocking (O_NONBLOCK): η read() επιστρέφει αμέσως· αν δεν υπάρχουν δεδομένα γυρνά −1 με errno == EAGAIN/EWOULDBLOCK.
2 τρόποι επικοινωνίας μεταξύ processes
  • Pipes / Named pipes (FIFOs) — μονόδρομο κανάλι bytes μεταξύ συγγενικών (ή μη) διεργασιών.
  • Sockets — αμφίδρομα, και μεταξύ διαφορετικών μηχανών.
  • Άλλες αποδεκτές: shared memory, message queues, signals, κοινά αρχεία.
Hard links vs Soft links — 2 διαφορές
  • Hard link: δείχνει στο ίδιο inode — το αρχείο ζει όσο υπάρχει τουλάχιστον ένα hard link. Δεν επιτρέπεται σε directories, ούτε across filesystems.
  • Soft (symbolic) link: ξεχωριστό αρχείο που περιέχει ένα path. Μπορεί να διασχίζει filesystems και να δείχνει σε φάκελο, αλλά «σπάει» (dangling) αν διαγραφεί ο στόχος.
TCP vs UDP — 2 διαφορές
  • TCP: connection-oriented, αξιόπιστο — εγγυάται παράδοση και σωστή σειρά (stream of bytes).
  • UDP: connectionless, best-effort — datagrams που μπορεί να χαθούν ή να φτάσουν εκτός σειράς, αλλά με μικρότερο overhead.
Stack vs Heap — διαφορά
  • Stack: αυτόματη δέσμευση (τοπικές μεταβλητές), LIFO, ελευθερώνεται μόλις επιστρέψει η συνάρτηση, περιορισμένο μέγεθος.
  • Heap: δυναμική δέσμευση με malloc()/free(), ζει μέχρι να την ελευθερώσεις ρητά, μεγαλύτερο αλλά πιο αργό.
  • Σύνδεση με threads: κάθε thread έχει δικό του stack αλλά όλα μοιράζονται το ίδιο heap.
2 τρόποι να περάσεις παραμέτρους μέσω exec()
  • Ορίσματα γραμμής εντολών — ο πίνακας argv που δίνεις στην execvp(file, argv) γίνεται το argv της main() του νέου προγράμματος.
  • Μεταβλητές περιβάλλοντος — μέσω execle()/execve() (όρισμα envp) ή κληρονομιά του environ· το νέο πρόγραμμα τις διαβάζει με getenv().
🧠
Μοτίβο απάντησης για «2 διαφορές»
Δώσε έναν άξονα σύγκρισης ανά διαφορά και τις δύο πλευρές του: «Ως προς το address space: η process …, ενώ το thread …». Έτσι ο διορθωτής βλέπει αμέσως ότι είναι όντως διαφορά και όχι δύο τυχαία facts.

3. Άσκηση «Συνάρτηση με Sockets» — Λυμένη

Η τυπική άσκηση sockets είναι μια συνάρτηση C που διαβάζει δυαδικά δεδομένα από socket, τα επεξεργάζεται και γράφει το αποτέλεσμα πίσω. Τα δύο σημεία που βαθμολογούνται: σωστό byte order (ntoh*/hton*) και σωστό read/write loop (η read/write μπορεί να μεταφέρει λιγότερα bytes απ' όσα ζήτησες!).

Άσκηση 1 — recv_and_sum

Γράψτε μια συνάρτηση int recv_and_sum(int sock) σε C/C++ που υλοποιεί τον εξής αλγόριθμο:

  • i) Από το socket sock λαμβάνεται πρώτα ένας ακέραιος (4 bytes, network byte order) που αντιστοιχεί στο πλήθος n των επόμενων αριθμών.
  • ii) Έπειτα λαμβάνονται n ακέραιοι (2 bytes ο καθένας, επίσης network byte order).
  • iii) Υπολογίζεται το άθροισμά τους.
  • iv) Το αποτέλεσμα στέλνεται πίσω στον αποστολέα (network byte order) μέσω του ίδιου socket.
  • v) Η συνάρτηση επιστρέφει 0 σε περίπτωση επιτυχίας, −1 σε περίπτωση σφάλματος.

Ερώτημα (β): Αν ο αποστολέας στείλει πρώτα n = 5 και μετά τους αριθμούς 1, 2, 3, 4, 5, ποια τιμή (σε δεκαδικό) θα σταλεί πίσω;

// Βοηθητική: διαβάζει ΑΚΡΙΒΩΣ n bytes — η read() μπορεί να γυρίσει λιγότερα
ssize_t read_all(int fd, void *buf, size_t n) {
    size_t total = 0;
    while (total < n) {
        ssize_t r = read(fd, (char*)buf + total, n - total);
        if (r <= 0) return -1;   // 0: έκλεισε η σύνδεση, -1: σφάλμα
        total += r;
    }
    return total;
}

// Βοηθητική: γράφει ΑΚΡΙΒΩΣ n bytes — συμμετρική της read_all
ssize_t write_all(int fd, const void *buf, size_t n) {
    size_t total = 0;
    while (total < n) {
        ssize_t w = write(fd, (const char*)buf + total, n - total);
        if (w <= 0) return -1;
        total += w;
    }
    return total;
}

int recv_and_sum(int sock) {
    // (i) πλήθος n: 4 bytes σε network byte order → uint32_t + ntohl
    uint32_t n_net;
    if (read_all(sock, &n_net, 4) != 4) return -1;
    uint32_t n = ntohl(n_net);

    // (ii)+(iii) n αριθμοί των 2 bytes → uint16_t + ntohs, αθροίζονται
    uint32_t sum = 0;
    for (uint32_t i = 0; i < n; i++) {
        uint16_t v_net;
        if (read_all(sock, &v_net, 2) != 2) return -1;
        sum += ntohs(v_net);
    }

    // (iv) αποτέλεσμα πίσω σε network byte order
    uint32_t sum_net = htonl(sum);
    if (write_all(sock, &sum_net, 4) != 4) return -1;

    return 0;   // (v) επιτυχία
}
Απάντηση ερωτήματος (β) — γιατί 15

Η συνάρτηση αθροίζει τους 5 αριθμούς: 1+2+3+4+5 = 15. Το htonl() αλλάζει τη σειρά των bytes στο καλώδιο, όχι την τιμή — ο παραλήπτης που θα κάνει ntohl() θα δει πάλι 15. Αν η ερώτηση ήταν «τι θα δει παραλήπτης που ξεχνάει το ntohl σε little-endian μηχανή», η απάντηση θα ήταν το byte-swapped 15 = 0x0F000000 = 251.658.240 — αυτό ακριβώς το λάθος ψαρεύει η άσκηση.

🎯
Το checklist της άσκησης socket
1️⃣ Σωστό μέγεθος τύπου: 4 bytes → uint32_t/int + ntohl, 2 bytes → uint16_t/short + ntohs.
2️⃣ Loop γύρω από read/write — μην υποθέσεις ότι ένα read φέρνει όλα τα bytes.
3️⃣ Έλεγχος κάθε κλήσης και return -1.
4️⃣ Μετατροπή και στην αποστολή (htonl πριν το write) — το μισό byte order είναι μηδέν βαθμοί byte order.
Άσκηση 2 — παραλλαγή: get_mydata

Γράψτε int get_mydata(int in_socket, int out_socket, char *st) που: i) διαβάζει από το in_socket 2 bytes που θεωρούμε ακέραιο n· ii) πολλαπλασιάζει κάθε byte του πίνακα st με το n και το γράφει ως ακέραιο (4 bytes) στο out_socket, μέχρι να συναντήσει byte ίσο με 0 στο st — το μηδενικό στέλνεται κι αυτό.

Ερώτημα (β): Αν το st έχει συνολικά 122 bytes (μαζί με το μηδενικό), πόσα bytes θα γραφτούν στο out_socket;

int get_mydata(int in_socket, int out_socket, char *st) {
    uint16_t n_net;                       // 2 bytes → short
    if (read_all(in_socket, &n_net, 2) != 2) return -1;
    int n = ntohs(n_net);

    for (int i = 0; ; i++) {
        int32_t val = st[i] * n;          // byte × n → int
        int32_t val_net = htonl(val);
        if (write_all(out_socket, &val_net, 4) != 4) return -1;
        if (st[i] == 0) break;           // το 0 στάλθηκε ήδη (0×n = 0)
    }
    return 0;
}
Απάντηση ερωτήματος (β)
Κάθε ένα από τα 122 bytes (121 δεδομένα + το μηδενικό) γράφεται ως ακέραιος των 4 bytes: 122 × 4 = 488 bytes. Το μοτίβο της ερώτησης: «πόσα bytes βγαίνουν» = πλήθος εγγραφών × μέγεθος εγγραφής στην έξοδο — όχι το μέγεθος εισόδου.

4. Άσκηση «Συγχρονισμένα Threads» — Λυμένη

Το βαρύτερο και πιο προβλέψιμο μοτίβο: N threads πρέπει να ενεργήσουν με αυστηρή σειρά, υποχρεωτικά με pthread_mutex_t + pthread_cond_t, ρητά χωρίς sleep() και χωρίς busy waiting. Η λύση είναι πάντα το ίδιο σχήμα: μια κοινή μεταβλητή «σειράς» (turn) + while + cond_wait + broadcast.

Άσκηση 1 — εκτύπωση με σειρά A→B→C

Γράψτε πρόγραμμα C που δημιουργεί 3 threads, τα οποία εκτελούν κυκλικά: Thread A τυπώνει "A", Thread B τυπώνει "B", Thread C τυπώνει "C". Η εκτύπωση πρέπει να γίνει με τη σειρά A → B → C → A → B → C… για 9 συνολικά εκτυπώσεις (A-B-C-A-B-C-A-B-C).

Απαραίτητα: χρήση pthread_cond_t και pthread_mutex_t για τον συντονισμό· να μη χρησιμοποιηθεί sleep() ή busy waiting· η λύση να είναι απολύτως συγχρονισμένη.

#include <stdio.h>
#include <pthread.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  cond = PTHREAD_COND_INITIALIZER;
int turn = 0;                          // 0 → A, 1 → B, 2 → C : ποιανού είναι η σειρά

void *worker(void *arg) {
    int id = *(int *)arg;
    const char *names[] = {"A", "B", "C"};

    for (int i = 0; i < 3; i++) {      // 3 κύκλοι × 3 threads = 9 εκτυπώσεις
        pthread_mutex_lock(&lock);

        while (turn != id)              // while, ΟΧΙ if! (spurious wakeups)
            pthread_cond_wait(&cond, &lock);

        printf("%s\n", names[id]);

        turn = (turn + 1) % 3;          // δίνω τη σειρά στο επόμενο
        pthread_cond_broadcast(&cond);  // ξυπνάω ΟΛΟΥΣ — βλ. γιατί παρακάτω
        pthread_mutex_unlock(&lock);
    }
    return NULL;
}

int main(void) {
    pthread_t tids[3];
    int ids[3] = {0, 1, 2};           // ΞΕΧΩΡΙΣΤΟ storage ανά thread (όχι &i!)

    for (int i = 0; i < 3; i++)
        pthread_create(&tids[i], NULL, worker, &ids[i]);
    for (int i = 0; i < 3; i++)
        pthread_join(tids[i], NULL);

    return 0;
}
Τα 4 σημεία που κρίνουν τον βαθμό
  • 1. while γύρω από το cond_wait — αν βάλεις if, ένα spurious wakeup ή «λάθος» ξύπνημα τυπώνει εκτός σειράς.
  • 2. broadcast αντί για signal — το signal ξυπνάει ένα τυχαίο thread· αν ξυπνήσει το λάθος, ξανακοιμάται και κανείς δεν ξυπνά το σωστό → deadlock. (Εναλλακτικά: ένα condvar ανά thread και στοχευμένο signal — επίσης σωστό.)
  • 3. &ids[i] και όχι &i — αν δώσεις &i σε όλα, μέχρι να τρέξουν ίσως δουν όλα i == 3.
  • 4. Η εκτύπωση γίνεται μέσα στο κλείδωμα — εδώ θέλουμε αυστηρή σειρά, άρα και το printf προστατεύεται.
Αυτόματο μηδέν στην άσκηση
while (turn != id) { /* τίποτα */ } χωρίς cond_wait = busy waiting — απαγορεύεται ρητά από την εκφώνηση. Το ίδιο και sleep(1) «για να προλάβει το άλλο thread». Η μόνη αποδεκτή αναμονή είναι pthread_cond_wait (που ελευθερώνει το mutex όσο κοιμάται και το ξανακλειδώνει στο ξύπνημα).
Άσκηση 2 — παραλλαγή με N threads σε αλυσίδα

Πρόγραμμα με 10 threads· καθένα παίρνει ως όρισμα το id του (1–10). Κάθε thread τυπώνει τον αριθμό του και μετά «δίνει τη σκυτάλη» στο επόμενο (το 1 στο 2, το 2 στο 3, …)· το thread 10 δεν ειδοποιεί κανένα.

Πώς γενικεύεται — ίδιος σκελετός
  • Η ίδια λύση με turn από 1 έως 10: while (turn != my_id) pthread_cond_wait(...).
  • Μετά την εκτύπωση: turn++ και broadcast — το «σήμα στο επόμενο» είναι η αλλαγή του turn + το ξύπνημα.
  • Το thread 10 απλώς δεν αυξάνει το turn (ή το αυξάνει σε τιμή που δεν περιμένει κανείς) — δεν χρειάζεται ειδική περίπτωση στο condvar.
  • Αν ζητηθεί «κάθε thread τυπώνει μέχρι να ειδοποιηθεί», το μοτίβο μένει ίδιο: η συνθήκη του while γίνεται while (!my_turn_to_stop).

Μάθε έναν σκελετό (mutex + turn + while/wait + broadcast) και προσαρμόζεις μόνο τη συνθήκη. Είναι το ίδιο μοτίβο με μια blocking ουρά εργασιών: while (ουρά_άδεια && !shutdown) pthread_cond_wait(...) — δες τη γέφυρα με τις εργασίες.

5. Άσκηση «Fork + Exec + Wait» — Λυμένη

Η άσκηση processes ζητάει σχεδόν πάντα τον ιερό συνδυασμό: fork() → στο παιδί exec*() → στον γονιό wait()/waitpid(). Είναι ο πυρήνας του πώς ένας job scheduler (όπως αυτός των εργασιών) εκτελεί εντολές — εδώ στη «γυμνή» εκδοχή της εξέτασης.

Άσκηση 1 — εκτέλεση εντολής σε παιδί

Γράψτε πρόγραμμα C/C++ που: παίρνει ως ορίσματα γραμμής εντολών 3 strings (π.χ. ./prog hello world again). Κατόπιν δημιουργεί ένα παιδί με fork(). Το παιδί καλεί execvp() για να τρέξει το /bin/echo με τις παραμέτρους που δόθηκαν. Ο γονιός περιμένει να τελειώσει το παιδί και τυπώνει "Child done".

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

int main(int argc, char *argv[]) {
    if (argc != 4) {                              // πρόγραμμα + 3 strings
        fprintf(stderr, "Usage: %s s1 s2 s3\n", argv[0]);
        exit(1);
    }

    pid_t pid = fork();
    if (pid == -1) { perror("fork"); exit(1); }

    if (pid == 0) {                               // ── παιδί ──
        char *args[] = {"echo", argv[1], argv[2], argv[3], NULL};
        execvp("/bin/echo", args);
        perror("execvp");                         // εδώ φτάνουμε ΜΟΝΟ αν αποτύχει
        _exit(1);
    }

    waitpid(pid, NULL, 0);                       // ── γονιός: περιμένει ──
    printf("Child done\n");
    return 0;
}
Τα σημεία που βαθμολογούνται
  • args[0] = όνομα προγράμματος (σύμβαση), τελευταίο στοιχείο NULL — χωρίς το NULL η execvp έχει undefined behavior.
  • Η exec δεν επιστρέφει ποτέ σε επιτυχία — αντικαθιστά ολόκληρη την εικόνα της διεργασίας. Κώδικας μετά την execvp τρέχει μόνο σε αποτυχία.
  • Το wait εγγυάται τη σειρά εξόδου: πρώτα το echo του παιδιού, μετά το "Child done". Χωρίς wait, η σειρά θα ήταν απρόβλεπτη (και το παιδί θα γινόταν zombie αν τερμάτιζε πρώτο).
  • _exit() στο παιδί μετά από αποτυχία exec — αποφεύγει double-flush των stdio buffers του γονιού.
Άσκηση 2 — παραλλαγή: Fibonacci με διαδοχικά παιδιά

Πρόγραμμα που παίρνει από το command line έναν ακέραιο ν. Δημιουργεί ν διαδοχικά παιδιά με fork()· κάθε παιδί (όχι ο γονιός) τυπώνει έναν όρο της σειράς Fibonacci (F0=0, F1=1, Fn=Fn−1+Fn−2).

Το κρυφό insight της άσκησης Fibonacci

Μετά το fork, γονιός και παιδί έχουν ξεχωριστά αντίγραφα μνήμης. Ό,τι υπολογίσει το παιδί χάνεται όταν τερματίσει — άρα τη σειρά πρέπει να την προχωράει ο γονιός, και κάθε παιδί απλώς τυπώνει την τρέχουσα τιμή:

long a = 0, b = 1;
for (int i = 0; i < n; i++) {
    pid_t pid = fork();
    if (pid == 0) {              // παιδί: τυπώνει ΚΑΙ τερματίζει
        printf("F%d = %ld\n", i, a);
        _exit(0);                 // αλλιώς θα συνέχιζε το loop!
    }
    waitpid(pid, NULL, 0);       // "διαδοχικά" = ένα-ένα με τη σειρά
    long next = a + b; a = b; b = next;   // ο ΓΟΝΙΟΣ προχωράει τη σειρά
}

Δύο κλασικά λάθη: (1) να ξεχάσεις το _exit() στο παιδί → κάθε παιδί συνεχίζει το loop και κάνει δικά του fork → έκρηξη διεργασιών· (2) να υπολογίζει το παιδί τον επόμενο όρο → ο γονιός δεν το βλέπει ποτέ.

6. Άσκηση «Bash Script» — Λυμένη

Η άσκηση bash έχει πάντα την ίδια «χορογραφία»: έλεγχος πλήθους παραμέτρων → έλεγχος ύπαρξης/δικαιωμάτων αρχείου ή καταλόγου → grep/wc → φιλικό μήνυμα. Αν ξέρεις τα test flags (-f -r -d) και τη διαφορά grep -c vs grep -o | wc -l, η άσκηση είναι 15 λεπτά.

Άσκηση 1 — μέτρηση λέξης-κλειδί σε αρχείο

Γράψτε ένα bash script που δέχεται 2 παραμέτρους: ένα αρχείο κειμένου και μία λέξη-κλειδί. Το πρόγραμμα πρέπει να ελέγξει αν το αρχείο υπάρχει και είναι αναγνώσιμο. Αν ναι, θα μετρά και θα τυπώνει πόσες φορές εμφανίζεται η λέξη-κλειδί στο αρχείο (ολόκληρη λέξη, όχι υπολέξη). Αν όχι, θα εμφανίζει κατάλληλο μήνυμα σφάλματος.

#!/bin/bash

# 1. Έλεγχος πλήθους παραμέτρων
if [ $# -ne 2 ]; then
    echo "Usage: $0 <file> <keyword>"
    exit 1
fi

file=$1
word=$2

# 2. Υπάρχει (ως κανονικό αρχείο) ΚΑΙ είναι αναγνώσιμο;
if [ ! -f "$file" ] || [ ! -r "$file" ]; then
    echo "Error: '$file' does not exist or is not readable"
    exit 1
fi

# 3. Μέτρηση εμφανίσεων ΟΛΟΚΛΗΡΗΣ λέξης (-w), μία ανά γραμμή εξόδου (-o)
count=$(grep -ow "$word" "$file" | wc -l)

if [ "$count" -gt 0 ]; then
    echo "'$word' appears $count times in '$file'"
else
    echo "'$word' not found in '$file'"
fi
✗ Η παγίδα
grep -c "$word" "$file"
Μετράει γραμμές που περιέχουν τη λέξη — αν μια γραμμή την έχει 3 φορές, μετράει 1. Επίσης χωρίς -w το «cat» ταιριάζει και στο «catalog».
✓ Το σωστό
grep -ow "$word" "$file" | wc -l
Το -o τυπώνει κάθε εμφάνιση σε δική της γραμμή, το -w ταιριάζει μόνο ολόκληρες λέξεις, το wc -l τις μετράει.
Tο toolbox των ελέγχων
Test Σημασία
[ $# -ne 2 ] πλήθος παραμέτρων ≠ 2
[ -f "$x" ] υπάρχει & είναι κανονικό αρχείο
[ -r "$x" ] / [ -w ] / [ -x ] αναγνώσιμο / εγγράψιμο / εκτελέσιμο
[ -d "$x" ] υπάρχει & είναι κατάλογος
[ -e "$x" ] υπάρχει (οτιδήποτε)

Πάντα quotes γύρω από τις μεταβλητές ("$file") — αλλιώς ονόματα με κενά σπάνε το test, και είναι από τα πρώτα πράγματα που κοιτάει ο διορθωτής.

Άσκηση 2 — παραλλαγή: αναζήτηση σε κατάλογο

Script που δέχεται ακριβώς 3 παραμέτρους: έναν κατάλογο, μια φράση και ένα αρχείο εξόδου. Ελέγχει ότι ο κατάλογος υπάρχει, ψάχνει τη φράση στα αρχεία του καταλόγου και καταγράφει στο αρχείο εξόδου σε ποια αρχεία και πόσες φορές εμφανίζεται.

#!/bin/bash
# 1. Έλεγχος πλήθους παραμέτρων
if [ $# -ne 3 ]; then
    echo "Usage: $0 <dir> <phrase> <outfile>"
    exit 1
fi

dir=$1; phrase=$2; out=$3

# 2. Έλεγχος ότι είναι κατάλογος
if [ ! -d "$dir" ]; then
    echo "Error: '$dir' is not a directory"
    exit 1
fi

# 3. Άδειασμα/δημιουργία αρχείου εξόδου
> "$out"

# 4. Σάρωση των αρχείων του καταλόγου
for f in "$dir"/*; do
    [ -f "$f" ] || continue      # αγνόησε υποκαταλόγους κλπ.
    count=$(grep -o "$phrase" "$f" | wc -l)
    if [ "$count" -gt 0 ]; then
        echo "$f: $count" >> "$out"
    fi
done
🧠
Ο σκελετός που λύνει κάθε bash άσκηση
1️⃣ if [ $# -ne N ] → usage + exit 1.  2️⃣ test ύπαρξης (-f/-r ή -d) → error + exit 1.  3️⃣ Η «δουλειά» (grep/wc/for).  4️⃣ Φιλικό μήνυμα. Γράψε τα 4 βήματα ως σχόλια πρώτα και γέμισέ τα.

7. Drill: «Τι θα Τυπώσει;»

Ερωτήσεις τύπου «τι τυπώνει / πόσες φορές / τι τιμή επιστρέφει» είναι ιδανικές για multiple choice και mini ερωτήματα. Δούλεψε τα παρακάτω στο χαρτί πρώτα και μετά αποκάλυψε την απάντηση.

🔥 Fork & Processes

Πόσες φορές τυπώνεται το "hi";
int main() { fork(); fork(); printf("hi\n"); }
4 φορές. Το 1ο fork → 2 διεργασίες· καθεμιά κάνει το 2ο fork → 4 διεργασίες· όλες φτάνουν στο printf. Γενικά: n σειριακά fork → 2ⁿ διεργασίες.
Πόσες φορές τυπώνεται το "hi";
for (int i = 0; i < 3; i++) fork();
printf("hi\n");
8 φορές (2³ διεργασίες). Κάθε iteration διπλασιάζει τις διεργασίες που τρέχουν το loop — και ΟΛΕΣ (γονείς και παιδιά) συνεχίζουν μετά το loop στο printf.
Τι θα τυπωθεί;
printf("A"); // χωρίς \n
fork();
printf("B\n");
"AB" δύο φορές (π.χ. AB ↵ AB). Το "A" δεν έχει \n, μένει στον stdio buffer· το fork αντιγράφει και τον buffer· κάθε διεργασία τελικά κάνει flush "AB\n". Κλασική παγίδα του buffering!
Τι θα τυπώσει το παιδί και τι ο γονιός;
int x = 5;
pid_t pid = fork();
if (pid == 0) { x = 10; printf("child x=%d\n", x); }
else { wait(NULL); printf("parent x=%d\n", x); }
child x=10 και parent x=5. Μετά το fork η μνήμη είναι ξεχωριστή (copy-on-write) — η αλλαγή του παιδιού δεν φαίνεται στον γονιό. Αντίθεση με τα threads, που μοιράζονται τα globals. Αυτό είναι και το insight της άσκησης Fibonacci στην §5.
Στην άσκηση fork+exec της §5 (fork → παιδί execvp echo → γονιός waitpid → printf "Child done"): μπορεί το "Child done" να εμφανιστεί ΠΡΙΝ από την έξοδο του echo;
Όχι. Το waitpid() μπλοκάρει μέχρι να τερματίσει το παιδί — και το παιδί τερματίζει αφού το echo ολοκληρώσει την έξοδό του. Χωρίς το wait όμως, η σειρά θα ήταν απρόβλεπτη (race μεταξύ γονιού/παιδιού στον scheduler).

🧵 Threads & Συγχρονισμός

Δύο threads κάνουν από 1.000.000 φορές counter++ σε global counter, ΧΩΡΙΣ mutex. Τι τιμή θα έχει στο τέλος ο counter;
Απρόβλεπτη, συνήθως < 2.000.000. Το counter++ είναι 3 βήματα (load → add → store)· όταν διαπλέκονται, χάνονται ενημερώσεις (lost update). Σωστή απάντηση σε multiple choice: «οποιαδήποτε τιμή από 2 έως 2.000.000» — όχι «πάντα 2.000.000» και όχι «πάντα 1.000.000».
Στη λύση A→B→C, τι θα πάθαινε το πρόγραμμα αν αντί για while (turn != id) γράφαμε if (turn != id);
Μετά από spurious wakeup (ή broadcast που ξυπνά «λάθος» thread) το thread θα συνέχιζε χωρίς να ξαναελέγξει τη συνθήκη → θα τύπωνε εκτός σειράς. Κανόνας: το cond_wait μπαίνει ΠΑΝΤΑ σε while που ξαναελέγχει τη συνθήκη.
Στην ίδια λύση, γιατί pthread_cond_broadcast και όχι pthread_cond_signal;
Το signal ξυπνάει ένα αυθαίρετο από τα threads που περιμένουν. Αν ξυπνήσει αυτό που ΔΕΝ έχει σειρά, θα δει turn != id και θα ξανακοιμηθεί — και το σωστό thread δεν θα ξυπνήσει ποτέ → deadlock. Το broadcast τα ξυπνάει όλα και «φιλτράρει» το while. (Εναλλακτικά σωστό: ένα condvar ανά thread + στοχευμένο signal.)
Τι κάνει «αυτόματα» η pthread_cond_wait(&cond, &lock) με το mutex;
Τρία πράγματα ατομικά: (1) ελευθερώνει το lock, (2) κοιμίζει το thread στο cond, (3) όταν ξυπνήσει, ξανακλειδώνει το lock πριν επιστρέψει. Γι' αυτό η συνθήκη μπορεί να ξαναελεγχθεί με ασφάλεια στο while.

🔌 I/O, Sockets & Bash

Η read() σε socket επέστρεψε 0. Τι σημαίνει;
EOF — η άλλη πλευρά έκλεισε τη σύνδεση. ΔΕΝ είναι σφάλμα (το σφάλμα είναι −1). Ένα σωστό helper τύπου read_line/read_all ελέγχει ακριβώς αυτό ξεχωριστά: «return 0 → ο πελάτης αποσυνδέθηκε».
Ζήτησες write(fd, buf, 1000) και επέστρεψε 600. Είναι σφάλμα; Τι κάνεις;
Όχι, είναι «short write» — νόμιμη συμπεριφορά σε sockets/pipes. Συνεχίζεις να γράφεις τα υπόλοιπα 400 από το buf+600 σε loop — αυτό ακριβώς κάνει μια βοηθητική write_all(), και είναι βαθμολογούμενο σημείο στις ασκήσεις socket.
Τι θα τυπωθεί στην ΟΘΟΝΗ;
int fd = open("out.txt", O_WRONLY|O_CREAT|O_TRUNC, 0644);
dup2(fd, 1);
printf("hello\n");
Τίποτα. Το dup2(fd, 1) κάνει το fd 1 (stdout) αντίγραφο του fd του αρχείου — το "hello" πάει στο out.txt. Είναι το κλασικό μοτίβο ανακατεύθυνσης της εξόδου ενός job σε αρχείο: dup2(out_fd, STDOUT_FILENO) πριν την execvp.
Το αρχείο data.txt περιέχει μία γραμμή: cat catalog cat. Τι τυπώνει το grep -c "cat" data.txt και τι το grep -ow "cat" data.txt | wc -l;
grep -c → 1 (μετράει γραμμές που ταιριάζουν). grep -ow | wc -l → 2 (εμφανίσεις ολόκληρης λέξης: τα 2 σκέτα "cat"· το "catalog" αποκλείεται από το -w). Αν έλειπε το -w θα ήταν 3. Αυτή η διάκριση είναι το κλασικό ζητούμενο των bash ασκήσεων.
Γράφεις σε socket του οποίου η άλλη πλευρά έχει κλείσει. Τι σήμα παίρνει η διεργασία σου και πώς προστατεύεσαι;
SIGPIPE — και η default συμπεριφορά είναι να ΣΚΟΤΩΣΕΙ τη διεργασία! Προστασία: signal(SIGPIPE, SIG_IGN) ώστε η write να επιστρέφει απλώς −1 με errno == EPIPE. Συνηθισμένη πρώτη γραμμή της main σε κάθε server.

8. Από τις Εργασίες στην Εξέταση

«Αν γράψατε τον κώδικα των εργασιών, θα είστε μια χαρά» — να γιατί. Κάθε οικογένεια ασκήσεων αντιστοιχεί σε μια ιδέα που υλοποίησες στις εργασίες (κονσόλα ↔ συντονιστής ↔ workers που εκτελούν jobs). Δεν έχει σημασία πώς ονόμασες τα αρχεία ή τις συναρτήσεις σου — σημασία έχει το μοτίβο. Πριν την εξέταση, ξαναπέρνα αυτές τις 6 ιδέες:

Οικογένεια άσκησης Η ιδέα στις εργασίες Τι να προσέξεις
Socket function Το πρωτόκολλο επικοινωνίας κονσόλας ↔ συντονιστή: βοηθητικές που γράφουν/διαβάζουν όλα τα bytes ή μια ολόκληρη γραμμή, με προσυμφωνημένη μορφή μηνύματος. Το while-loop γύρω από read/write, ο έλεγχος r <= 0, το νόημα του read() == 0
Byte order Το γέμισμα της sockaddr_in: το port μπαίνει πάντα με htons(port), οι σταθερές διευθύνσεις με htonl(). Πότε hton* (πριν φύγει στο δίκτυο) και πότε ntoh* (μόλις έρθει)
Condvar συγχρονισμός Η ουρά εργασιών: workers που περιμένουν με cond_wait όσο η ουρά είναι άδεια, push που κάνει signal, shutdown που κάνει broadcast για να ξυπνήσουν όλοι. Γιατί while και όχι if· γιατί το shutdown θέλει broadcast ενώ για ένα νέο στοιχείο αρκεί signal
Fork / exec / wait Η εκτέλεση ενός job: fork → ανακατεύθυνση εξόδου με dup2 → execvp → waitpid και ανάγνωση του exit code με WIFEXITED/WEXITSTATUS. argv με NULL στο τέλος, perror + _exit ΜΕΤΑ την execvp, dup2 πριν την exec
Signals Suspend/resume εργασιών με kill(pid, SIGSTOP/SIGCONT), handlers που απλώς σηκώνουν ένα volatile sig_atomic_t flag, και SIG_IGN στο SIGPIPE στους servers. Ο handler μόνο σηκώνει flag· SIGSTOP/SIGCONT = πάγωμα/συνέχιση· γιατί αγνοούμε το SIGPIPE
Processes vs Threads Η ίδια η εξέλιξη του project: από processes + pipes σε threads + sockets — και γιατί τα κοινά δεδομένα χρειάστηκαν ξαφνικά mutexes. Κοινός address space → προστασία με mutex· ξεχωριστός → επικοινωνία μόνο με IPC
Blocking pop από κοινή ουρά — ο σκελετός του συγχρονισμού
// Worker περιμένει δουλειά: το μοτίβο condvar
pthread_mutex_lock(&m);
while (queue_is_empty && !shutdown)   // η συνθήκη ΠΑΝΤΑ σε while
    pthread_cond_wait(&cond, &m);     // κοιμάται ΧΩΡΙΣ να κρατά το mutex
/* ... παίρνω στοιχείο με κλειδωμένο mutex ... */
pthread_mutex_unlock(&m);

Άλλαξε τη συνθήκη του while σε turn != id και έχεις τη λύση της εκτύπωσης A→B→C. Ίδιο μοτίβο, άλλη συνθήκη.

Εκτέλεση εντολής σε παιδί — ο σκελετός του fork/exec
// Πώς ένας scheduler τρέχει ένα job
pid_t pid = fork();
if (pid == 0) {
    dup2(out_fd, STDOUT_FILENO);   // προαιρετική ανακατεύθυνση εξόδου
    execvp(argv[0], argv);         // argv: NULL στο τέλος!
    _exit(1);                      // μόνο σε αποτυχία exec
}
waitpid(pid, &status, 0);
int code = WIFEXITED(status) ? WEXITSTATUS(status) : -1;

Στην εξέταση συνήθως λείπει το dup2 και προστίθεται «τύπωσε Child done» — ευκολότερο από αυτό που ήδη παρέδωσες.
